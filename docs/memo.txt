まだ残っているブロッカー

WebSocket join が壊れる可能性が高い

ここは直した方がいい。

RealtimeClient.emit() は payload を object として spread する実装になっている。 しかし app 側では join-space に currentSpace.id という文字列を渡している。

つまり、実際に送られる JSON は期待される

{"action":"joinSpace","spaceId":"..."}

ではなく、文字列を spread したような { "0": "...", "1": "...", "action": "joinSpace" } 形になり、body.spaceId が入らない可能性が高い。WebSocket handler は body.spaceId を前提に join / broadcast している。

修正は小さい。

newSocket.emit('join-space', { spaceId: currentSpace.id })

ついでに RealtimeClient.emit() 側も防御的にしておくとよい。

emit(event, payload = {}) {
  const action = socketEventToAction(event)
  if (!action || ws.readyState !== WebSocket.OPEN) return
  const body =
    payload && typeof payload === 'object'
      ? payload
      : event === 'join-space'
        ? { spaceId: payload }
        : {}
  ws.send(JSON.stringify({ action, ...body }))
}

なお、テストでは client.emit('join-space', { spaceId: 'space-1' }) で検証しているため、現行 app 側の文字列呼び出しは拾えていない。

既存DB移行なし方針について

それなら、かなり楽になる。既存DB移行を前提にした Ticket 09 / migrate script は実運用パスから外してよい。

docs では Ticket 09 が「必要であれば」となっており、既存 SQLite / uploads を AWS に移すスクリプトを用意する扱いになっている。 さらに reservation でも「新規DBで始めてもよい。その場合、このチケットは後回しにできる」と書かれている。

今回の方針なら、Ticket 09 は「後回し」ではなく not planned / archive でよい。AWS 版は clean DynamoDB で始める前提にする。

追加で気になるが、ブロッカーではない点

WebSocket の $disconnect は session を消すだけで、残りの接続へ session-left や session count 更新を broadcast していない。 join 時は session-joined と space-joined-successfully を送っているので、入室方向だけ通知される。

これはデプロイ不能ではないが、複数人接続時の session count / 鍵交換状態が少し stale になる可能性がある。最初は polling fallback を残すなら致命傷ではない。
