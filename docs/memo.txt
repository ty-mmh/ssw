確認した。結論としては、方向性はかなり良い。ただし、まだ AWS に deploy してよい段階ではない。特に Lambda 起動時に落ちる可能性が高い箇所 がある。

総評

Codex / Claude Code の対応で、以前の「Raspberry Pi 常駐 Express + Socket.IO + SQLite + local uploads」から、かなり明確に AWS サーバレス寄りへ切られている。

良い変更としては、AWS SDK 依存が追加され、DynamoDB / S3 / API Gateway Management API へ寄せる準備が入っている。socket.io 依存も package から外れている。

server.js もかなり整理されていて、store / service / route の境界ができ、以前の Socket.IO 常駐ロジックは消えている。

合言葉の平文保存をやめて HMAC hash 化する方向も入っている。createSpace では passphraseHash を作って保存し、レスポンスから passphrase / passphraseHash を落とす sanitizeSpace も入っている。

DynamoDB / S3 / WebSocket / Budget まで CDK に入っているのも良い。DynamoDB は Spaces, Messages, Connections に分かれていて、messages / connections には TTL が入っている。 S3 uploads bucket も public block / encryption / SSL / lifecycle 付き。 Budget guard も入っている。

最優先で直すべき点

1. Lambda が better-sqlite3 require で落ちる可能性が高い

ここが最大のブロッカー。

storage/create-stores.js が top-level で better-sqlite3 を require している。 一方で CDK の Lambda bundling では better-sqlite3 を external にしている。

つまり、Lambda では STORAGE_DRIVER=dynamodb なので SQLite を使わないつもりでも、require('../storage/create-stores') した時点で better-sqlite3 が必要になり、cold start で Cannot find module 'better-sqlite3' になる可能性が高い。

修正方針は、better-sqlite3 を SQLite 分岐内へ遅延 require すること。

// NG: top-level
const Database = require('better-sqlite3')

// OK: sqlite branch only
if (storageDriver !== 'dynamodb') {
  const Database = require('better-sqlite3')
  db = db || new Database(env.SQLITE_DB_PATH || 'secure_chat.db')
}

これは即修正でよい。

2. Static hosting のデプロイが未完成に見える

CDK では static bucket と CloudFront distribution は作られている。 ただ、public/ を S3 にアップロードする BucketDeployment 的な定義が見当たらない。

このままだと、インフラは立っても CloudFront の origin bucket が空で、アプリ本体が配信されない可能性がある。

追加するなら：

const s3deploy = require('aws-cdk-lib/aws-s3-deployment')

new s3deploy.BucketDeployment(this, 'DeployStaticAssets', {
  sources: [s3deploy.Source.asset(path.join(repoRoot, 'public'))],
  destinationBucket: staticBucket,
  distribution,
})

3. S3 直接アップロード用 CORS が足りない

フロントは presigned POST を受け取り、ブラウザから S3 に直接 fetch(presign.uploadUrl, { method: 'POST', body: formData }) している。 S3 側も presigned POST を返す実装になっている。

ただ、uploads bucket 定義に CORS が見当たらない。 ブラウザ直アップロードは CORS で詰まる可能性が高い。

最小なら uploads bucket にこれを足す。

cors: [{
  allowedMethods: [s3.HttpMethods.POST, s3.HttpMethods.GET],
  allowedOrigins: ['*'],
  allowedHeaders: ['*'],
}]

後で CloudFront domain に絞ればよい。

4. WebSocket URL がフロントに渡っていない

CDK は WebSocket API を作って output している。 しかし、フロントの runtime-config.js は wsUrl: '' のまま。 /api/config も PUBLIC_WS_URL を返す設計だけど、HTTP Lambda の environment に PUBLIC_WS_URL が入っていない。

なので、現状 AWS 版では WebSocket が有効化されず、polling fallback で動く形になるはず。これは「段階移行」としてはあり。ただし WebSocket まで動かしたいなら CDK で以下を追加する必要がある。

httpFunction.addEnvironment('PUBLIC_WS_URL', websocketStage.url)

HTTP API を CloudFront 経由にするなら PUBLIC_API_BASE_URL は空でもよい。

5. WebSocket 経由の media 受信が S3 で壊れる

通常の履歴取得では、S3 key を presigned download URL に解決してから fetch している。

一方、WebSocket で image/audio を受けた時は receivedMessage.encrypted_content をそのまま fetch している。 AWS 版では encrypted_content に S3 key が入るため、この経路だけメディア表示に失敗する。

ここは WebSocket 受信側でも resolveDownloadUrl 相当を使う必要がある。

6. presign API の入力検証が弱い

Express route 側では /presign-upload に spaceId / messageId 必須チェックがある。 しかし Lambda HTTP handler では body をそのまま createPresignedUpload に渡している。

また /presign-download は任意の key を受けて署名URLを返す。 S3 key はランダムなので即致命的ではないが、少なくとも spaces/... prefix のみ許可する、.. や空文字を弾く、可能なら message record と照合する、くらいは入れたい。

運用上の注意

APP_SECRET はローカルでは default secret に落ちる。 AWS では /ssw/app-secret の secure SSM parameter を読む想定になっている。

移行スクリプトも getAppSecret() を使って既存 passphrase を hash 化する。 なので、Raspberry Pi の既存データを AWS に移す時は、必ず AWS 本番と同じ APP_SECRET で dry-run / migration する必要がある。ここを間違えると、移行後に既存の合言葉で入れなくなる。

あと、scripts/setup-database.js は既存 DB を削除する。 Raspberry Pi 本番環境で誤って打つと危険なので、ファイル名か警告を強めてもよい。
